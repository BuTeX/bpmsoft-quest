# Архитектура BPMSoft Quest

Актуально на 2026-07-21.

## 1. Технологии

- HTML, CSS и JavaScript без клиентского сборщика;
- Node.js HTTP-сервер;
- PostgreSQL или временное memory-хранилище;
- пароли `scrypt`, непрозрачные 30-дневные сессии;
- `HttpOnly`/`SameSite` cookie;
- четыре локальных кэша в `localStorage` и серверная синхронизация;
- локальные изображения и шрифты.

## 3. Граница данных и представления

Исполняемая логика правильных ответов хранится в JavaScript:

- `app.js` — первая карта;
- `chapter2-missions.js` — вторая карта;
- `chapter3-missions.js` — третья карта.
- `chapter4-missions.js` — четвёртая карта.

`index.html` содержит стартовые значения и тексты, которые видны до первого динамического рендера. При переработке контента нужно синхронно обновлять данные и fallback-тексты.

Технические идентификаторы, ключи состояния, имена функций и файлов ассетов не являются видимым каноном. Их не нужно переименовывать вместе с текстами, если нет миграции и отдельной технической задачи.

## 4. Модель задания Академии

Объект в `missions` обычно содержит:

- ключ, номер, область, заголовок и режим;
- completion-флаг, XP, время и следующую разблокировку;
- краткое описание, вводные, сценарий, принцип и источник;
- инструменты, правильную конфигурацию и варианты;
- справочные термины;
- сообщения успеха и повторного прохождения;
- данные интерактивной сцены, если они нужны режиму.

Режимы:

| Режим | Задания | Поведение |
|---|---|---|
| `sequence` | 01, 04 | правильный порядок уникальных элементов |
| `matching` | 02, 05 | назначение терминов требованиям |
| `quiz` | 03, 07 | последовательные вопросы |
| `gateway` | 06 | сборка интеграции и три теста |
| `classification` | 08 | 12 требований и повторно используемые категории |
| `blueprint` | 09 | шесть узлов решения и три приёмочных сценария |

Интерактивные renderer-функции используют прежние внутренние имена (`renderForgeScene`, `renderAccessScene`, `renderEngineScene`, `renderNodeScene` и другие). Это реализация механики, а не названия в UI.

## 5. Модель заданий 10–27

Вторая и третья карты используют данные `mission → phases → slots → options`.

Каждый slot содержит:

- идентификатор;
- короткую рабочую метку;
- вопрос;
- правильный идентификатор варианта;
- объяснение;
- правдоподобные варианты ответа.

`chapter2.js` и `chapter3.js`:

- рендерят активный этап;
- сохраняют выбранные ответы;
- перемешивают варианты;
- блокируют неполную проверку;
- фиксируют верные решения после частичной ошибки;
- очищают неверные решения;
- уменьшают число оставшихся попыток;
- повторяют только активный этап после исчерпания попыток;
- завершают задание после последнего этапа.

Обычное задание третьей карты содержит четыре этапа по три решения, итоговое — пять этапов.

## 6. Модель заданий 28–36

Четвёртая карта использует данные `mission → stages → hotspots/cards/solution`. После осмотра трёх точек игрок заполняет роли `cause`, `mechanism`, `test`. Карточки поддерживают клик и drag-and-drop; ошибка сохраняет верные звенья и помечает слабые.

## 7. Состояние и канонизация

Ключи локального кэша:

```text
bpmsoft-quest-v1
bpmsoft-quest-chapter2-v1
bpmsoft-quest-chapter3-v1
bpmsoft-quest-chapter4-v1
```

Каждая карта хранит XP, уровень, оставшиеся попытки, активное задание, временные ответы, просмотренные вводные, completion-флаги и финальное достижение.

Канонизация выводит XP и доступное задание из непрерывной цепочки completion-флагов. Сохранение с разрывом не открывает более позднее задание. Менее продвинутое состояние не перезаписывает более высокий серверный прогресс, кроме явного сброса.

Канонические XP:

- Академия: `55, 100, 150, 205, 250, 350, 400, 450, 500`;
- АО «Медные машины»: `50, 100, 150, 200, 250, 300, 350, 400, 500`;
- группа «Семь дорог»: `60, 120, 180, 240, 300, 360, 420, 480, 600`.
- сеть «Золотая полка»: `70, 140, 210, 280, 350, 420, 490, 560, 700`.

## 8. Сервер

Основные маршруты:

- регистрация, вход, выход и получение текущего аккаунта;
- чтение и изменение режима пользователя;
- `GET/PUT/DELETE /api/account/progress/chapter1|chapter2|chapter3|chapter4`;
- проверка админ-пароля;
- healthcheck;
- публичная раздача разрешённых корневых файлов и `assets/`.

Сервер:

- нормализует email;
- хранит salted `scrypt`-хеш;
- сравнивает чувствительные значения без раннего выхода;
- очищает незнакомые поля состояния;
- запрещает запись основного прогресса в режиме изучения;
- обновляет `updated_at` и поддерживает миграцию локального кэша в пустой аккаунт.

## 9. Правила добавления или изменения задания

1. Сначала проверить [`CONTENT_AND_MISSION_RULES.md`](CONTENT_AND_MISSION_RULES.md).
2. Подтвердить факты документацией BPMSoft.
3. Зафиксировать роль, симптом, исходные данные и критерий приёмки.
4. Обновить объект задания и fallback-тексты.
5. При изменении прогрессии обновить initial state, completion-флаги, канонические XP и серверный sanitizer.
6. Обновить карту, разблокировку и next-action.
7. Добавить или изменить тест.
8. Проверить desktop, mobile, клавиатуру и reduced motion.

Текстовая переработка не даёт права менять правильный ответ. Если новый сценарий требует другой модели решения, это отдельное продуктово-техническое изменение.

## 9. Арты

- карты: `1600 × 1000`;
- панорамы: `1600 × 500`;
- смысловые объекты должны читаться без встроенного текста;
- нижняя часть панорамы остаётся спокойной для HTML-overlay;
- канонические промпты находятся в `assets/image-prompts.md`;
- официальный логотип не генерируется внутри игровых иллюстраций.
