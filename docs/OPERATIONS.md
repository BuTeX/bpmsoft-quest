# Эксплуатация и откат

## Контроль перед релизом

1. Зафиксировать версию приложения и хеш коммита.
2. Выполнить `npm ci`, `npm test` и `npm run test:e2e`.
3. Создать проверенный backup командой `BACKUP_DIR=... DATABASE_URL=... ./scripts/backup-postgres.sh`.
4. Выполнить restore drill в отдельной одноразовой БД через `RESTORE_DATABASE_URL=... BACKUP_FILE=... ./scripts/restore-drill.sh`.
5. Проверить `/ready` и авторизованный `/metrics` после выкладки.

Закрытый pilot-preview запускается с `REQUIRE_PRODUCTION_READINESS=false`. Перед публичным запуском заполните все mail/legal/metrics-переменные из `.env.example`, включите `REQUIRE_PRODUCTION_READINESS=true` и повторите readiness-проверку.

Миграции в проекте только additive: они добавляют таблицы, столбцы и индексы, но не удаляют данные. Сервер применяет их до открытия HTTP-порта и фиксирует в `schema_migrations`.

## Наблюдаемость

- `/health` — состояние процесса и доступность БД;
- `/ready` — readiness probe для платформы;
- `/metrics` — Prometheus-метрики запросов, задержек и uptime; в production требуется `Authorization: Bearer $METRICS_TOKEN`;
- `LOG_FORMAT=json` — структурированный журнал запросов с request ID, статусом и длительностью без email и содержимого ответов;
- заголовок `X-Request-ID` связывает клиентскую ошибку с серверным журналом.

Минимальные алерты:

- `/ready` возвращает не `200` две проверки подряд;
- доля `5xx` выше 1% за 5 минут;
- p95 длительности API выше 1 секунды за 10 минут;
- backup не создавался более 25 часов;
- restore drill не выполнялся более 30 дней.

## Откат

1. Остановить дальнейшее продвижение релиза.
2. Сохранить логи, request ID и текущий дамп БД.
3. Откатить приложение на предыдущий подтверждённый артефакт/коммит.
4. Не откатывать additive-миграции: предыдущая версия должна игнорировать новые таблицы и столбцы.
5. Восстанавливать БД из backup только при подтверждённой порче данных и после явного решения владельца данных. Restore выполняется в отдельной БД, проверяется, затем переключается строка подключения.
6. После отката повторить smoke E2E, `/ready`, вход, сохранение прогресса и админ-аналитику.

Целевые показатели для пилота: RPO не более 24 часов, RTO не более 2 часов. Перед широким запуском владелец сервиса должен подтвердить более строгие значения или принять эти.
