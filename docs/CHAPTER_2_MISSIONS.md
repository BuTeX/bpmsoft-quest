## Mission 10. Sorting Furnace: поставка двойников

| Поле | Значение |
| --- | --- |
| Номер | 10 |
| Уровень | III — Внешний контур |
| Режим | `data-pipeline` |
| Награда | 50 XP карты |
| Откроет | Portal Gate |
| Проверяет | импорт Excel, сопоставление, справочники, ключи дублей, контроль фоновой загрузки |

### Сюжетное вступление

**Название:** «Плавка одинаковых лиц».

Сортировочная домна работает без остановки, но каждая новая партия содержит всё больше одинаковых големов. Смотритель `SORT-3` докладывает о рекордном выпуске и считает это доказательством успеха.

ГЕФЕСТ-7 замечает, что на корпусах двойников выбит один и тот же внешний номер заказа. Архитектор должен остановить поток до того, как дубли займут все склады.

### Ситуация BPMSoft

В раздел «Производственные заказы» повторно загружается Excel-файл внешней системы. В файле есть:

- `ExternalOrderId` — устойчивый внешний идентификатор;
- название заказчика;
- производственный район — справочное поле;
- приоритет;
- плановая дата;
- комментарий, который иногда остаётся пустым.

Часть заказов уже существует и должна обновиться, остальные должны быть созданы. В значениях района встречаются лишние кавычки и специальные символы.

### Этап 1. Проверка поставки

Игрок выбирает подготовительные действия:

- сверить права пользователя, выполняющего импорт;
- нормализовать справочные значения района;
- убедиться, что внешний идентификатор заполнен;
- не создавать заранее все варианты района из файла вслепую.

### Этап 2. Сопоставление колонок

Каждая колонка файла назначается полю объекта. `ExternalOrderId` обязательно сохраняется в отдельное поле записи и не подменяется названием заказчика.

### Этап 3. Правило дублей

Правильный ключ — `ExternalOrderId`.

Если игрок добавляет к нему название заказчика, интерфейс поясняет, что несколько выбранных колонок работают через логическое **И**. Изменение названия тогда может привести к созданию новой записи вместо обновления существующей.

### Этап 4. Три партии

1. Новый идентификатор создаёт одну запись.
2. Существующий идентификатор обновляет заполненные в Excel поля.
3. Повтор той же партии не создаёт второго заказа; пустой комментарий не очищает уже заполненное значение.

Импорт завершается в фоне. Игрок находит уведомление и при необходимости понимает, что выполнение контролируется через журнал процессов и процесс `FileImportProcess`.

### Каноническая модель решения

1. Проверить файл, права и качество ключевого поля.
2. Нормализовать справочные значения до импорта.
3. Сопоставить колонки с правильными полями объекта.
4. Использовать устойчивый внешний идентификатор как условие поиска дубля.
5. Запустить загрузку и проверить её результат через уведомление/журнал процессов.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| искать дубль только по названию заказчика | название может меняться и не является устойчивым идентификатором сценария |
| выбрать `ExternalOrderId + Название` | условия объединяются через И; изменившееся название разрушит совпадение |
| оставить справочники как есть | кавычки и специальные символы могут создать новые элементы справочника |
| ожидать, что пустая ячейка очистит поле | при обновлении пустые значения Excel не очищают заполненные поля BPMSoft |
| считать запуск мастера доказательством успеха | импорт выполняется в фоне и должен быть проверен |

### Результат в мире

Домна переплавляет корпуса двойников. Ковчег Форм начинает выпускать разные детали согласно реальным заказам. `SORT-3` меняет показатель с «числа выпущенных корпусов» на «число подтверждённых уникальных заказов».

## Mission 11. Portal Gate: печати чужого цеха

| Поле | Значение |
| --- | --- |
| Номер | 11 |
| Уровень | III — Внешний контур |
| Режим | `access-matrix` |
| Награда | 50 XP карты |
| Откроет | Signal Yard |
| Проверяет | конфигурацию портала, организации, функциональные роли, доступ к записям и полям |

### Сюжетное вступление

**Название:** «Ворота без меры».

После первых сообщений об утечке смотритель `PORTA-5` запечатал Портальные ворота для всех. Законные заказчики не могут узнать состояние своих поставок, а ремонтные гильдии отправляют запросы голубиной почтой.

Полная блокировка безопасна только для самого смотрителя. Архитектору нужно вернуть доступ, не позволяя одной гильдии видеть заказы другой.

### Ситуация BPMSoft

Две внешние организации — «Латунь» и «Сланец» — работают через клиентский портал.

Требования:

- сотрудник видит и создаёт заказы своей организации;
- не видит заказы другой организации;
- видит публичную стоимость, но не внутреннюю себестоимость;
- администратор организации управляет пользователями своей портальной организации;
- отключённый пользователь не может войти;
- внутренний диспетчер работает в основном приложении и видит обе организации согласно своей роли.

### Этап 1. Выбор пространства

Игрок выбирает клиентский портал как внешнее пространство для работы клиентов с собственными записями и пользовательским разделом заказов.

### Этап 2. Организации и роли

Для каждого контрагента создаётся портальная организация, связанная с контрагентом. Контакты добавляются как отдельные портальные пользователи. Администратор получает функциональную роль организации, а не глобальные права основного приложения.

### Этап 3. Матрица доступа

Игрок настраивает четыре слоя:

1. доступ к портальному разделу;
2. права по записям своей организации;
3. ограничение внутренней себестоимости на уровне поля;
4. отдельная внутренняя роль диспетчера.

Для уже существующих заказов после изменения правил требуется актуализация прав доступа. Это отдельное действие, потому что новые правила автоматически применяются только к новым записям.

### Этап 4. Проверка ворот

1. Пользователь «Латуни» открывает свой заказ — разрешено.
2. Он пробует открыть заказ «Сланца» — отказ.
3. Администратор «Латуни» приглашает коллегу — разрешено в рамках организации.
4. Отключённая учётная запись пытается войти — отказ.
5. Внутренняя себестоимость скрыта для обоих внешних пользователей.

### Каноническая модель решения

Портал определяет внешнее пространство, организация объединяет пользователей контрагента, функциональная роль задаёт полномочия администратора организации, права по записям разделяют данные организаций, права по колонкам скрывают чувствительное поле. Существующие записи получают новую модель после актуализации прав.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| один общий портал без разделения организаций | пространство само по себе не ограничивает записи разных клиентов |
| поле «Контрагент» без правил доступа | значение поля не является механизмом авторизации |
| скрыть себестоимость только в макете страницы | скрытый элемент интерфейса не заменяет права на колонку |
| дать администратору организации системную роль администратора | это чрезмерные полномочия основного приложения |
| изменить правила и не актуализировать существующие записи | новые правила применяются автоматически только к новым записям |

### Результат в мире

Шагающий Курьер открывает по отдельному шлюзу для каждой гильдии. Печати на воротах становятся бело-оранжевыми: белая часть обозначает организацию, оранжевая — разрешённое действие.

### Источники

- `portal/administrirovanie-organizatsii-i-polzovateley-na-portale.md`;

## Mission 12. Signal Yard: контракт релейных башен

| Поле | Значение |
| --- | --- |
| Номер | 12 |
| Уровень | III — Внешний контур |
| Режим | `contract-console` |
| Награда | 50 XP карты |
| Откроет | Cycle Foundry |
| Проверяет | настройку исходящего REST-сервиса, Basic-аутентификацию, системные настройки и обработку ответа |

### Сюжетное вступление

**Название:** «Сигнал с пустой печатью».

Релейные башни исправно отправляют сообщения, но внешняя мастерская отвечает одним красным импульсом. Смотритель `RELAY-12` усиливает мощность передатчика, хотя проблема явно не в дальности.

На контрольной пластине ГЕФЕСТ-7 видит `401`. После последнего переноса релейная башня сохранила форму печати, но потеряла её секретное значение.

### Контракт сценария

Внешняя мастерская предоставляет REST-метод:

```text
POST /v1/dispatch
HTTPS
Basic authentication
request: externalOrderId, workshopCode, priority
response: dispatchId, status
recommended timeout for this contract: 5000 ms
```

Успешный ответ должен записать `dispatchId` и `status` в заказ. Вызов выполняется из бизнес-процесса после подтверждения заказа.

### Этап 1. Карточка сервиса

Игрок задаёт базовый HTTPS URI, код сервиса и создаёт REST-метод `POST` с путём `/v1/dispatch`.

### Этап 2. Запрос и ответ

Поля заказа сопоставляются параметрам запроса, а `dispatchId` и `status` — параметрам ответа. `GET`, пустое тело или запись всего ответа в комментарий не выполняют контракт.

### Этап 3. Аутентификация

Логин хранится в системной настройке типа «Строка», пароль — в системной настройке типа «Зашифрованная строка». Метод использует Basic-аутентификацию только через HTTPS.

Настройки привязываются к пакету интеграции, но их значения по умолчанию не переносятся между средами. После установки пакета секреты нужно ввести на целевой среде.

### Этап 4. Контрольные вызовы

1. Корректный заказ получает `200`, `dispatchId` сохраняется.
2. Неверный пароль возвращает `401`; запись не помечается отправленной.
3. После переноса пакета пустая системная настройка обнаруживается до продуктивного запуска.
4. Превышение `5000 ms` обрабатывается как ошибка вызова, а не как успешная отправка.

### Каноническая модель решения

Создать исходящий REST-сервис и метод согласно контракту; сопоставить запрос и ответ; хранить Basic-учётные данные в системных настройках, пароль — зашифрованной строкой; использовать HTTPS; после переноса среды заполнить значения секретов; вызывать метод из процесса и явно обрабатывать отказ.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| записать логин и пароль непосредственно в процесс | секреты должны поступать из системных настроек |
| хранить пароль обычной строкой | для пароля предусмотрена «Зашифрованная строка» |
| считать Base64 шифрованием | Basic безопасно применять только поверх HTTPS |
| ожидать перенос значения секрета вместе с пакетом | значение по умолчанию системной настройки не сохраняется при переносе |
| помечать заказ отправленным до успешного ответа | ошибка или тайм-аут тогда создадут ложное состояние |

### Результат в мире

Башни отвечают друг другу короткими подтверждёнными импульсами. Тягач Магистралей прокладывает оранжевый маршрут к северному кольцу.

### Источники

- `integrations/nachalo-raboty-s-integraciej-veb-servisov.md`;
- `integrations/nastroit-autentifikatsiyu-veb-servisa.md`;
- `platform/biznes-protsessy.md`.

## Mission 13. Cycle Foundry: смена, которая не заканчивается

| Поле | Значение |
| --- | --- |
| Номер | 13 |
| Уровень | III — Внешний контур |
| Режим | `process-trace` |
| Награда | 50 XP карты |
| Откроет | Package Depot |
| Проверяет | актуальную версию процесса, журналирование, сериализацию, максимальное число повторений |

### Сюжетное вступление

**Название:** «Вечная ночная смена».

Один и тот же вагон проходит через Цех циклов в сорок седьмой раз. Рабочие големы честно повторяют проверку и каждый раз отправляют его к началу маршрута.

Новая схема процесса уже висит в диспетчерской, но трасса на полу повторяет старый путь. Чтобы остановить конвейер, нужно определить, какая версия действительно исполняется.

### Ситуация BPMSoft

У процесса согласования производственного заказа есть:

- версия `2.4`, помеченная актуальной;
- исправленная версия `2.5`, сохранённая, но не актуальная;
- ветвление, которое при пустом результате проверки возвращает заказ на повтор;
- отсутствующее ограничение числа повторений;
- длинное ожидание действия пользователя;
- журналирование включено только у процесса, но не у диагностически важного элемента проверки.

### Этап 1. Чтение трассы

Игрок сопоставляет экземпляр из журнала с версией `2.4` и видит повторяющийся участок. Наличие сохранённой `2.5` не доказывает её выполнение: система запускает актуальную версию.

### Этап 2. Диагностический контур

Нужно:

- включить журналирование процесса и элемента проверки;
- включить сериализацию для длительного процесса с ожиданием;
- задать максимальное число повторений;
- не включать системный контекст безопасности без отдельной необходимости сценария.

### Этап 3. Управляемая версия

Исправленная `2.5` получает признак актуальной версии. Старая схема сохраняется для истории, но новые экземпляры запускаются по `2.5`.

### Этап 4. Повторный запуск

1. Успешная проверка завершает маршрут.
2. Ошибка возвращает заказ на повтор не более установленного предела.
3. Длительное ожидание восстанавливается после перезапуска с сохранёнными параметрами.
4. Журнал показывает, где и почему был выбран путь повторения.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| только нарисовать исправление в новой версии | выполняется актуальная версия, а не последняя по номеру |
| удалить старую версию | она нужна для понимания уже запущенных экземпляров и истории |
| включить журналирование абсолютно всех элементов навсегда | это создаёт лишний объём; нужны диагностически значимые точки |
| ограничиться журналом без предела повторов | видимость не устраняет бесконечный цикл |
| считать сериализацию заменой журналированию | сериализация сохраняет состояние выполнения, журналирование показывает ход исполнения |

### Результат в мире

Кольцевой конвейер раскрывается в прямой маршрут. `CYCLE-8` впервые произносит номер следующей смены и передаёт Архитектору пластину актуальной версии.

### Источники

- `platform/biznes-protsessy.md`.

## Mission 14. Package Depot: краны противоречий

| Поле | Значение |
| --- | --- |
| Номер | 14 |
| Уровень | III — Внешний контур |
| Режим | `dependency-crane` |
| Награда | 50 XP карты |
| Откроет | уровень IV и Trace Furnace |
| Проверяет | состав пакетов, зависимости, расширение поставочных схем, перенос и привязанные данные |

### Сюжетное вступление

**Название:** «Контейнер, который держит сам себя».

Краны Пакетного депо застыли над платформами. Один контейнер зависит от второго, второй — от третьего, а третий требует первый. Ни один смотритель не решается опустить груз.

Среди контейнеров ГЕФЕСТ-7 находит три версии директивы `Δ-47`. Именно отсюда противоречие распространилось по Медному Пределу.

### Набор пакетов сценария

- `FactoryBase` — поставочный пакет платформы;
- `CopperCommon` — общие объекты, справочники и системные настройки проекта;
- `CopperPortal` — портальный раздел заказов;
- `CopperSignal` — исходящий веб-сервис и процесс отправки;
- `CopperFrontier` — прикладная логика карты.

`CopperCommon`, `CopperPortal`, `CopperSignal` и `CopperFrontier` — условные учебные имена пакетов внутри сценария «Медный Предел». Это не встроенные пакеты и не специальные термины BPMSoft. В интерфейсе миссии назначение этих пакетов повторяется непосредственно в условии каждого вопроса.

### Этап 1. Дерево зависимостей

Каноническая схема:

```text
FactoryBase
└─ CopperCommon
   ├─ CopperPortal
   ├─ CopperSignal
   └─ CopperFrontier
      ├─ depends on CopperPortal
      └─ depends on CopperSignal
```

Реализация может показать зависимости `CopperFrontier` двумя отдельными тросами. Циклические зависимости запрещены.

### Этап 2. Размещение артефактов

| Артефакт | Пакет |
| --- | --- |
| общий объект производственного заказа | `CopperCommon` |
| справочник районов и необходимые привязанные данные | `CopperCommon` |
| портальная страница/раздел | `CopperPortal` |
| карточка REST-сервиса и процесс отправки | `CopperSignal` |
| прикладная координация Медного Предела | `CopperFrontier` |

Поставочный `FactoryBase` не модифицируется напрямую. Изменения выполняются через зависимые заказные пакеты.

### Этап 3. Поставка

Игрок формирует перенос из разработки в тестовую среду:

- проверяет версии и зависимости;
- включает нужные схемы и привязанные данные;
- учитывает компиляцию серверных схем и применение изменений объектов;
- после установки вводит значения секретных системных настроек на целевой среде;
- проводит smoke-проверку портала, процесса и веб-сервиса.

### Контрольные ситуации

1. Кран обнаруживает цикл зависимостей — поставка блокируется.
2. Попытка изменить `FactoryBase` заменяется схемой расширения в заказном пакете.
3. Пакет устанавливается, но секрет пуст — интеграционный тест блокирует выпуск до заполнения настройки.
4. Справочник не был привязан — проверка фиксирует неполную поставку до продуктивной среды.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| положить все артефакты в один бесформенный пакет | усложняется перенос, повторное использование и управление зависимостями |
| изменить поставочный пакет | поставочную функциональность расширяют через зависимый заказной пакет |
| создать взаимную зависимость Portal ↔ Signal | циклические зависимости запрещены |
| ожидать, что все справочные данные перенесутся автоматически | необходимые данные должны быть явно привязаны к пакету |
| считать секрет частью переносимой конфигурации | значение настройки нужно указать на целевой среде повторно |

### Результат в мире

Краны выстраивают контейнеры в ясное дерево. Внешнее кольцо приходит в общий ритм, а створки внутреннего кольца открываются. На одной из пластин виден первый след административного изменения `Δ-47`.

3. выдал смотрителю функциональную роль;
4. процесс обновил приоритет заказа;
5. прямой SQL-скрипт изменил другое поле, но в журнале изменений этого события нет.

| считать отсутствие записи доказательством отсутствия изменения | прямой SQL может пройти мимо журнала изменений |
| дать всем право управлять журналом аудита | просмотр и управление разделены системными операциями |

## Mission 16. Change Assembly: выпуск без общего плана

| Поле | Значение |
| --- | --- |
| Номер | 16 |
| Уровень | IV — Внутренний контур |
| Режим | `release-board` |
| Награда | 50 XP карты |
| Откроет | Oracle Forge |
| Проверяет | связи проблемы, изменения, релиза, КЕ, команды и факта выпуска в BPMSoft Service |

### Сюжетное вступление

**Название:** «Девять ремонтов, ни одного выпуска».

Каждый мастер исправлял собственный цех и считал работу законченной. В Сборочной релизов лежат девять отдельных нарядов, но ни один не показывает, какие механизмы изменены вместе и кто отвечает за испытание.

Совет должен впервые оформить восстановление Сердца как одно управляемое изменение, а не как набор тайных ремонтов.

### Ситуация BPMSoft

Расследование выявило проблему: конфликтующие версии `Δ-47` вызывают повторную отправку заказов и потерю секретов после переноса.

Нужно:

- зарегистрировать проблему как причину повторяющихся инцидентов;
- создать изменение с целью, приоритетом, ответственным и затронутыми сервисами/КЕ;
- включить изменение в релиз;
- определить команду и этапы разработки, тестирования и развёртывания;
- связать требуемые активности;
- после успешного выпуска перевести релиз в состояние «Выпущен» и получить фактическую дату.

### Этап 1. Цепочка сущностей

Каноническая связь:

```text
Проблема → Изменение → Релиз
               ├─ Сервисы
               ├─ Конфигурационные единицы
               └─ Активности выполнения
```

Проблема описывает повторяющуюся причину. Изменение описывает управляемый переход к целевому состоянию. Релиз группирует изменения, которые выпускаются вместе.

### Этап 2. Состав изменения

Игрок выбирает затронутые КЕ: Портальные ворота, Релейный двор, Цех циклов и Сердце Контура. Для каждой назначаются работы и ответственные. Поле «Родительское изменение» используется только если крупный CR действительно разбивается на связанные дочерние изменения.

### Этап 3. План релиза

В релиз добавляется изменение, команда и этапы. Отдельные активности отвечают за разработку, тестирование и развёртывание. Игра не создаёт вымышленную коробочную сущность CAB: согласование моделируется доступными активностями и/или визированием.

### Этап 4. Завершение

До прохождения приёмки релиз нельзя пометить выпущенным. После состояния «Выпущен» фактическая дата заполняется автоматически и может стать точкой запуска уведомлений.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| создать только релиз без изменения | теряются цель, причина, затронутые элементы и выполнение конкретного изменения |
| хранить все работы в описании | активности и команда нужны для ответственности и контроля этапов |
| связать инцидент напрямую с релизом, пропустив проблему/изменение | теряется управляемая цепочка причины и исправления |
| придумать отдельную коробочную сущность CAB | в локальном источнике согласование строится через активности и/или визирование |
| поставить «Выпущен» до приёмки | состояние должно отражать фактический выпуск, а не намерение |

### Результат в мире

Девять нарядов складываются в единую сборочную пластину. Совет мастеров подписывает ответственность за выпуск, а Оранжевый шум впервые ослабевает.

## Mission 18. Contour Heart: синхронизация Медного Предела

| Поле | Значение |
| --- | --- |
| Номер | 18 |
| Уровень | IV — финальный boss |
| Режим | `contour-war-room` |
| Награда | 100 XP карты |
| Откроет | финал второй главы |
| Проверяет | сквозную диагностику, минимальное исправление, пакетирование, выпуск и приёмку |

### Сюжетное вступление

**Название:** «Девять голосов Сердца».

В центральной камере Сердце Контура говорит несколькими голосами. Один требует ускорить каждый заказ, второй блокирует внешние сигналы, третий снова запускает старую версию процесса. Все директивы когда-то были правильными, но относятся к разным состояниям системы.

Архитектор не может просто выключить Сердце: вместе с ним остановятся энергоячейки всего региона. Нужно доказать причину, провести согласованное изменение и перезапустить контур под нагрузкой.

### Инцидент

После переноса последнего выпуска наблюдаются четыре симптома:

1. пользователь организации «Латунь» видит старый заказ организации «Сланец»;
2. исходящий вызов мастерской возвращает `401`;
3. один подтверждённый заказ создаёт несколько отправок;
4. журнал процесса не показывает, на каком условии начался повтор.

Дополнительные факты:

- правила доступа изменены, но существующие записи не актуализированы;
- значение зашифрованной системной настройки на новой среде пусто;
- процесс `2.5` исправлен, но актуальной осталась версия `2.4` без предела повторов;
- диагностический элемент не журналируется;
- пакеты установлены без цикла, но checklist целевой среды не включал заполнение секрета и актуализацию прав.

### Фаза 1. Диагностика

Игрок связывает симптомы и причины:

| Симптом | Причина | Доказательство |
| --- | --- | --- |
| виден чужой старый заказ | права не актуализированы для существующих записей | правило действует на новые записи, старая запись сохранила прежние права |
| `401` после переноса | отсутствует значение системной настройки секрета | пакет сохранил настройку, но не её значение |
| несколько отправок | актуальна старая версия процесса без предела повторов | журнал экземпляра указывает версию `2.4` |
| нет условия повтора в трассе | элемент проверки не журналируется | в журнале есть вход и следующая отправка, но нет решения ветвления |

Ложные причины — «сломанный портал», «нужна новая LLM», «увеличить тайм-аут» и «дать всем полные права» — не соответствуют доказательствам.

### Фаза 2. Исправление и поставка

Минимальный согласованный набор:

1. актуализировать права доступа существующих записей;
2. заполнить зашифрованную системную настройку на целевой среде;
3. сделать исправленную версию `2.5` актуальной;
4. задать максимальное число повторений и журналирование проверочного элемента;
5. включить изменения в заказные пакеты без модификации поставочного слоя;
6. связать исправление с зарегистрированным изменением и релизом;
7. дополнить checklist целевой среды эксплуатационными действиями;
8. пройти тестирование до состояния релиза «Выпущен».

### Фаза 3. Четыре приёмочных сигнала

#### Сигнал A. Организации

Пользователь «Латуни» видит собственный новый и старый заказ, но не видит записи «Сланца». Внутренняя себестоимость скрыта.

#### Сигнал B. Авторизованная отправка

Подтверждённый заказ вызывает `POST /v1/dispatch`, получает `dispatchId` и сохраняет его один раз.

#### Сигнал C. Ошибка и повтор

При временной ошибке процесс выполняет контролируемый повтор в заданном пределе. Журнал показывает решение ветвления. Повтор не создаёт бесконечную цепочку отправок.

#### Сигнал D. Новая среда

Пакеты устанавливаются в порядке зависимостей; справочные данные присутствуют; секрет заполняется отдельно; права актуализируются; smoke-проверка блокирует выпуск, если любой шаг пропущен.

### Роль Оракула

Оракул отмечает рискованные сочетания и помогает выбрать порядок проверок, но его прогноз не является доказательством причины. Финальное решение опирается на конфигурацию, журналы и фактические приёмочные сигналы.

### Основные ловушки

| Ловушка | Почему неверно |
| --- | --- |
| выдать порталу полные права | скрывает симптом ценой утечки данных |
| увеличить тайм-аут для ответа `401` | код означает проблему аутентификации, а не ожидания |
| удалить процесс и собрать новый без журнала | теряется история и не доказана причина повторов |
| пометить релиз выпущенным до сигналов | состояние не соответствует фактической готовности |
| положиться только на прогноз Оракула | прогноз помогает приоритизировать, но не заменяет диагностику |
| объединить все пакеты в один | не требуется для исправления и ухудшает управляемость поставки |

### Финал миссии

После четвёртого сигнала девять голосов Сердца сходятся в один. Оранжевые каналы загораются от центра к внешнему кольцу, колоссальный фабричный голем поднимается и фиксирует общий ритм Медного Предела.
