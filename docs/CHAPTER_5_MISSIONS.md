## 2. Контентный контракт `operational-twin`

Каждый раунд содержит:

- от пяти до семи дискретных тактов `T0…Tn`;
- один первый подтверждённый checkpoint расхождения;
- не менее одного позднего симптома, который нельзя засчитать как первопричину;
- два или три управляемых правила с коротким набором допустимых значений;
- контрольный прогон с теми же входными событиями;
- позитивный исход и хотя бы один запрещённый промежуточный outcome;
- текстовый журнал, полностью дублирующий значимые изменения панорамы;
- ссылку на официальную возможность BPMSoft и отдельную маркировку проектных правил.

Правильное решение раунда:

```text
первый подтверждённый checkpoint + допустимая конфигурация + отсутствие запрещённых outcomes
```

Игрок не обязан угадывать внутреннюю причину до появления доказательства. Если нарушение правила происходит на `T2`, а заметный бизнес-симптом появляется на `T4`, правильным checkpoint остаётся `T2`.

Общие типы обратной связи:

- **Поздний симптом** — правило выбрано верно, но checkpoint поставлен после первого доказанного расхождения.
- **Неполная защита** — checkpoint верен, но повторный прогон всё ещё допускает запрещённое состояние.
- **Ложное улучшение** — конечный статус выглядит успешным, но по пути создан дубль, раскрыты лишние данные или отправлено преждевременное уведомление.
- **Подтверждённое восстановление** — исходные события не менялись, первое расхождение устранено, все наблюдаемые ограничения соблюдены.

## 3. Задание 37 — Центр расписания

### Цель

Научить отделять устойчивую идентичность операционного сегмента от отображаемого номера рейса и не позволять старому сообщению откатывать актуальную запись.

### Инициатор и симптом

Лада Ветрова сообщает: после изменения времени вылета сайт сначала показывает `10:40`, а через несколько минут возвращает `09:55`. В журнале есть две версии одного сегмента, полученные не в порядке их создания.

### Раунд A — Поздняя версия

| Такт | Подтверждённое событие |
|---|---|
| T0 | Сегмент `GA-417 / 21.07 / SIP→GRD` хранится в версии `3`, вылет `09:55`. |
| T1 | Система расписания публикует версию `4`, вылет `10:40`, время события `08:12`. |
| T2 | BPMSoft применяет версию `4`; сайт и обращение показывают `10:40`. |
| T3 | Партнёрский шлюз доставляет задержавшееся сообщение версии `3`, созданное в `08:05`; запись меняется на `09:55`. |
| T4 | Пассажиру формируется уведомление со старым временем. |

Первое расхождение: **T3**, когда заведомо более старая версия принята как актуальная. T4 — последствие.

Управляемые правила:

1. ключ записи: `рейс` / `рейс + дата` / `рейс + дата + сегмент`;
2. применение обновления: `последнее полученное` / `только версия выше текущей`;
3. отклонённое событие: `удалить` / `записать как отклонённое с причиной`.

Допустимая конфигурация: составной ключ сегмента, монотонное сравнение версии и журналирование отклонённого сообщения.

Запрещённые outcomes: вторая запись того же сегмента; откат времени; исчезновение следа отвергнутой версии.

### Раунд B — Код-шеринговый номер

| Такт | Подтверждённое событие |
|---|---|
| T0 | Операционный сегмент имеет стабильный ID `OP-882-21JUL`, номер «Гуд Авиа» `GA-882`. |
| T1 | Партнёр публикует маркетинговый номер `PA-204` для того же `OP-882-21JUL`. |
| T2 | Сопоставление по отображаемому номеру создаёт второй сегмент `PA-204`. |
| T3 | Одна пересадка связывается с `GA-882`, багаж — с `PA-204`. |
| T4 | Пассажир получает два разных статуса одного перелёта. |

Первое расхождение: **T2**, создание второй операционной сущности.

Управляемые правила: идентичность по стабильному ID операционного сегмента; маркетинговые номера как связанные значения, а не новые сегменты; негативная проверка одинакового номера в разные даты.

Позитивная приёмка: оба номера открывают один сегмент, а разные даты и участки маршрута не сливаются.

### Источники и граница

BPMSoft предоставляет объекты, поля и связи для проектной модели данных. Состав ключа авиационного сегмента, внешний ID и правила версий — **проектное решение «Гуд Авиа»**, а не заявленная коробочная модель рейса.

Основной источник: [BPMSoft — кастомизация No-code](https://edu.bpmsoft.ru/baza-znaniy/kastomizatsiya-no-code-1.0/?version=1.9).

## 4. Задание 38 — Центр пересадок

### Цель

Развести абсолютный момент времени, локальное представление аэропорта и рабочий календарь обслуживания.

### Инициатор и симптом

Руководитель центра пересадок сообщает: система считает часть ночных стыковок допустимыми, хотя второй сегмент уже отправлен по абсолютному времени; для обращений после закрытия смены неверно рассчитывается срок реакции.

### Раунд A — Переход даты и часового пояса

| Такт | Подтверждённое событие |
|---|---|
| T0 | Прибытие хранится как `2026-07-21 22:50 UTC`, аэропорт `UTC+3`. |
| T1 | Вылет следующего сегмента хранится как `2026-07-22 00:20 UTC`, аэропорт `UTC+5`. |
| T2 | Расчёт сравнивает подписи `01:50` и `05:20` без зон и даты, получая ложное окно `3 ч 30 мин`. |
| T3 | Пересадка отмечается доступной. |
| T4 | Пассажир получает недоступный вариант перебронирования. |

Первое расхождение: **T2**, сравнение локальных подписей вместо абсолютных моментов.

Управляемые правила: хранить расчётные даты в UTC; хранить часовой пояс аэропорта отдельно; показывать пользователю локальное время с датой и зоной.

Позитивная приёмка: расчёт выполняется по UTC, переход суток видим, а отображение каждого аэропорта остаётся локальным.

### Раунд B — SLA рабочего календаря

| Такт | Подтверждённое событие |
|---|---|
| T0 | Обращение о сорванной пересадке зарегистрировано в пятницу в `16:40` по локальному времени сервиса. |
| T1 | Договор задаёт реакцию `2 рабочих часа`, календарь — `09:00–18:00`, выходные суббота и воскресенье. |
| T2 | Срок рассчитывается как `18:40` пятницы по календарю `24×7`. |
| T3 | Система показывает просрочку, хотя рабочее время уже закончилось. |
| T4 | Руководитель получает ложную эскалацию. |

Первое расхождение: **T2**, выбор неверного календаря при расчёте срока.

Управляемые правила: выбирать правило по сервису/договору и приоритету; использовать календарь договора; показывать правило и календарь в расшифровке срока.

Запрещённые outcomes: расчёт по локальному времени браузера; смешение реакции и разрешения; скрытая подмена календаря на `24×7`.

### Источники и граница

Расчёт сроков реакции и разрешения по правилам, сервису, договору, приоритету и календарю — штатная возможность BPMSoft. Правило минимального времени пересадки и модель часовых поясов аэропортов — **проектное решение**.

Основной источник: [BPMSoft — время реакции и разрешения](https://edu.bpmsoft.ru/baza-znaniy/obrabotka-obrashcheniy/opredelenie-srokov-reaktsii-i-razresheniya/?version=1.9).

## 5. Задание 39 — Багажный узел

### Цель

Различить повтор одного события и новое событие, а также порядок создания события и порядок доставки.

### Инициатор и симптом

Координатор багажа сообщает: после сетевого таймаута создаются две задачи по одному чемодану, а поздний скан иногда возвращает статус с «Передан на пересадку» на «Погружен в исходном аэропорту».

### Раунд A — Повтор webhook

| Такт | Подтверждённое событие |
|---|---|
| T0 | Аэропорт отправляет событие `BAG-771/E17`: «Снят с рейса», сегмент `OP-882-21JUL`. |
| T1 | BPMSoft принимает событие и запускает задачу перенаправления, но ответ источнику задерживается. |
| T2 | Источник повторяет тот же `eventId=E17`; обработчик принимает его как новое событие. |
| T3 | Создаётся вторая задача перенаправления. |
| T4 | Два сотрудника одновременно меняют маршрут багажа. |

Первое расхождение: **T2**, повторный внешний ID принят как новое событие.

Управляемые правила: уникальность `источник + eventId`; повтор возвращает результат первой обработки; повтор не запускает второе бизнес-действие.

### Раунд B — События не по порядку

| Такт | Подтверждённое событие |
|---|---|
| T0 | Для багажа и сегмента применено событие с последовательностью `40`: «Разгружен». |
| T1 | Применено событие `42`: «Передан на пересадку». |
| T2 | Из очереди приходит задержавшееся событие `41`: «Погружен»; статус откатывается. |
| T3 | Система отменяет уже созданную задачу пересадки. |
| T4 | Багаж остаётся без актуального маршрута. |

Первое расхождение: **T2**, событие с меньшей последовательностью изменяет текущее состояние.

Управляемые правила: последовательность хранится на уровне `багаж + сегмент`; более старое событие остаётся в журнале, но не меняет текущий статус; новый сегмент начинает собственную последовательность.

### Источники и граница

Вебхуки BPMSoft могут принимать данные стороннего сервиса и запускать процесс или действие над объектом. Ключ идемпотентности, последовательность багажных событий и правила актуальности — **проектный интеграционный контракт**.

Основной источник: [BPMSoft — настройка интеграции вебхуков](https://edu.bpmsoft.ru/baza-znaniy/veb-servisy/nastroyka-integratsii-vebkhukov/?version=1.9).

## 6. Задание 40 — Центр нерегулярных перевозок

### Цель

Использовать кейс для вариативного жизненного цикла пассажирской ситуации и не создавать новый кейс на каждое сообщение.

### Инициатор и симптом

Руководитель клиентского сервиса сообщает: обращение закрывается сразу после выдачи первого предложения, хотя пассажир может его отклонить; сообщение из второго канала создаёт параллельное обращение с отдельным SLA.

### Раунд A — Преждевременное закрытие

| Такт | Подтверждённое событие |
|---|---|
| T0 | Создан кейс «Сорванная пересадка», стадия «Проверка маршрута». |
| T1 | Сотрудник формирует первый вариант и переводит ситуацию в «Ожидание решения пассажира». |
| T2 | Жёсткий линейный процесс автоматически закрывает обращение после отправки варианта. |
| T3 | Пассажир отклоняет вариант. |
| T4 | Ответ некуда зафиксировать, создаётся новое обращение. |

Первое расхождение: **T2**, закрытие до обязательного решения пассажира.

Управляемые правила: жизненный цикл задаётся стадиями кейса; закрытие разрешено только после подтверждённого результата; процесс автоматизирует шаги внутри стадии, но не подменяет вариативный маршрут кейса.

### Раунд B — Два канала, одна ситуация

| Такт | Подтверждённое событие |
|---|---|
| T0 | Открыт кейс для `пассажир + операционный сегмент + тип сбоя`. |
| T1 | Веб-форма добавляет сообщение к кейсу. |
| T2 | SMS с тем же сегментом создаёт второй открытый кейс. |
| T3 | Два SLA и два сотрудника запускают разные варианты перебронирования. |
| T4 | Пассажир получает противоречивые ответы. |

Первое расхождение: **T2**, второе сообщение ошибочно интерпретировано как новая ситуация.

Управляемые правила: искать открытый кейс по устойчивой бизнес-связке; добавлять коммуникацию в историю найденного кейса; новый кейс создавать только для новой ситуации или после закрытия предыдущей.

### Источники и граница

Стадии и шаги кейса — возможность BPMSoft. Бизнес-ключ пассажирской ситуации и правило объединения каналов — **проектное решение**.

Основной источник: [BPMSoft — настройка кейса](https://edu.bpmsoft.ru/baza-znaniy/lenta-stadiy-v-razdelakh/nastroyka-keysa/?version=1.9).

## 7. Задание 41 — Зал перебронирования

### Цель

Отделить автоматизированную подготовку решения от действия, требующего ответственности сотрудника, и отправлять коммуникацию только после подтверждённого внешнего результата.

### Инициатор и симптом

Руководитель нерегулярных перевозок сообщает: система автоматически подтверждает дорогой партнёрский маршрут и успевает уведомить пассажира до того, как партнёр подтвердил бронь.

### Раунд A — Граница решения

| Такт | Подтверждённое событие |
|---|---|
| T0 | Кейс подтверждает сорванную пересадку и собирает доступные варианты. |
| T1 | Правила ранжируют собственный рейс в тот же день и партнёрский рейс с ночёвкой. |
| T2 | Система без задачи сотруднику подтверждает партнёрский вариант с доплатой и ночёвкой. |
| T3 | Пассажир отклоняет изменение условий. |
| T4 | Требуется ручная отмена внешней брони. |

Первое расхождение: **T2**, высоковлияющее действие выполнено без подтверждения ответственной роли.

Управляемые правила: автоматизировать сбор и ранжирование; собственный эквивалентный маршрут можно проводить по утверждённому порогу; партнёрский/ночной/платный вариант создаёт задачу сотруднику.

### Раунд B — Уведомление до подтверждения

| Такт | Подтверждённое событие |
|---|---|
| T0 | Сотрудник подтверждает выбранный вариант. |
| T1 | Бизнес-процесс вызывает внешний сервис бронирования. |
| T2 | До ответа партнёра процесс отправляет пассажиру сообщение «Бронь подтверждена». |
| T3 | Внешний сервис возвращает отказ из-за отсутствия мест. |
| T4 | Кейс содержит подтверждение, которого не существует у перевозчика. |

Первое расхождение: **T2**, коммуникация утверждает ещё не подтверждённый результат.

Управляемые правила: отправлять финальное сообщение только по подтверждённому ответу; отказ переводит кейс в выбор альтернативы; ожидание/ошибка создаёт наблюдаемое состояние, а не ложный успех.

### Источники и граница

Бизнес-процессы BPMSoft поддерживают автоматические шаги, ветвления и пользовательские задачи; интеграции вызывают внешние сервисы. Порог допустимого автоматического перебронирования определяется бизнесом «Гуд Авиа».

Основной источник: [BPMSoft — бизнес-процессы](https://edu.bpmsoft.ru/baza-znaniy/biznes-protsessy/?version=1.9).

## 8. Задание 42 — Партнёрский терминал

### Цель

Проверить доступ не только по видимому списку, но и по прямой ссылке, колонкам и сроку временной передачи прав.

### Инициатор и симптом

Координатор партнёрской сети сообщает: сотрудник аэропорта видит только свои рейсы в реестре, но по прямой ссылке открывает пассажирскую запись другой организации; после окончания смены временные полномочия остаются активными.

### Раунд A — Ложная изоляция фильтром

| Такт | Подтверждённое событие |
|---|---|
| T0 | Пользователь партнёра A входит на портал; реестр отфильтрован по организации A. |
| T1 | У роли есть чтение объекта «Пассажирская ситуация». |
| T2 | Прямая ссылка на запись партнёра B открывается, включая финансовые колонки. |
| T3 | Пользователь выгружает лишние данные. |
| T4 | Аудит обнаруживает межорганизационный доступ. |

Первое расхождение: **T2**, фактическая проверка записи и колонок пропускает чужие данные. Фильтр на T0 не является защитой.

Управляемые правила: права по записи наследуются от организации партнёра; чувствительные колонки закрыты отдельно; негативный тест выполняется прямой ссылкой и API-запросом.

### Раунд B — Делегирование после срока

| Такт | Подтверждённое событие |
|---|---|
| T0 | Диспетчер делегирует заместителю права с `08:00` до `20:00`. |
| T1 | В `12:00` заместитель корректно получает рабочие задачи. |
| T2 | В `20:01` период закончился, но роли не актуализированы и доступ сохраняется. |
| T3 | Заместитель подтверждает действие после окончания смены. |
| T4 | В журнале действие выглядит формально разрешённым. |

Первое расхождение: **T2**, истёкшее делегирование продолжает действовать.

Управляемые правила: указывать начало и окончание; актуализировать роли после окончания/отзыва; проверять недоступность операции после срока.

### Источники и граница

BPMSoft поддерживает права по операциям, записям и колонкам, портальный принцип минимальных привилегий и делегирование с периодом действия. Привязка авиационной записи к организации партнёра — **проектная модель**.

Основные источники: [виды прав доступа](https://edu.bpmsoft.ru/baza-znaniy/polzovateli-i-prava-dostupa/vidy-prav-dostupa/?version=1.9), [права доступа на портале](https://edu.bpmsoft.ru/baza-znaniy/administrirovanie-portala/prava-dostupa-na-portale/?version=1.9), [передача прав доступа](https://edu.bpmsoft.ru/baza-znaniy/polzovateli-i-prava-dostupa/delegirovanie/?version=1.8).

## 9. Задание 43 — Интеграционная диспетчерская

### Цель

Сохранить сквозную корреляцию, отличить таймаут от подтверждённого отказа и безопасно повторить операцию с неизвестным исходом.

### Инициатор и симптом

Руководитель интеграций сообщает: входящее событие видно, но его нельзя связать с экземпляром процесса и исходящим запросом; после таймаута повтор иногда создаёт вторую бронь.

### Раунд A — Потерянная корреляция

| Такт | Подтверждённое событие |
|---|---|
| T0 | Webhook принимает `eventId=IRR-552`, `correlationId=C-908`. |
| T1 | Запускается процесс, связанный с рейсом и пассажирским кейсом. |
| T2 | Исходящий запрос создаёт новый случайный ID и не сохраняет `C-908`. |
| T3 | Получен таймаут; журнал показывает три несвязанные операции. |
| T4 | Поддержка повторяет весь сценарий, не зная состояния внешней брони. |

Первое расхождение: **T2**, корреляция теряется на границе исходящего вызова.

Управляемые правила: сохранять correlation ID в записи и параметрах процесса; передавать его исходящему вызову; связывать экземпляр процесса с бизнес-объектами и показывать статус/ошибку в журнале.

### Раунд B — Неизвестный исход и повтор

| Такт | Подтверждённое событие |
|---|---|
| T0 | Отправлена команда бронирования `operationKey=RB-771`. |
| T1 | Ответ не получен до таймаута; факт выполнения у партнёра неизвестен. |
| T2 | Повтор отправляется с новым ключом `RB-772`. |
| T3 | Партнёр подтверждает обе команды. |
| T4 | У пассажира появляются две брони. |

Первое расхождение: **T2**, повтор неизвестной операции меняет идемпотентный ключ.

Управляемые правила: таймаут означает `неизвестно`, а не `успех` или `отказ`; повтор использует тот же operation key; повтор разрешён только для классифицированной временной ошибки и имеет ограничение попыток.

Запрещённые outcomes: потеря correlation ID; новый бизнес-ключ на повторе; сообщение об успехе до подтверждения; бесконечный повтор.

### Источники и граница

BPMSoft предоставляет вебхуки, настройку исходящих REST/SOAP-сервисов, параметры процессов и журнал процессов. Формат correlation ID, ключ идемпотентности и политика повторов согласуются в **проектном интеграционном контракте**.

Основные источники: [интеграции No-code](https://edu.bpmsoft.ru/baza-znaniy/veb-servisy/?version=1.9), [журнал процессов](https://edu.bpmsoft.ru/baza-znaniy/soprovozhdenie-protsessov/razdel-zhurnal-protsessov/?version=1.9).

## 10. Задание 44 — Центр прогнозирования задержек

### Цель

Использовать прогноз как вход в проверяемый процесс, не подменять прогноз решением и не применять модель к данным, не прошедшим контроль качества.

### Инициатор и симптом

Руководитель операционной аналитики сообщает: модель оценивает вероятность задержки, после чего процесс автоматически перебронирует пассажиров; часть прогнозов построена по неполным данным нового маршрута.

### Раунд A — Прогноз не равен решению

| Такт | Подтверждённое событие |
|---|---|
| T0 | Для рейса собраны исторические и текущие признаки. |
| T1 | ML-модель возвращает вероятность задержки `78%` и версию модели `M12`. |
| T2 | Процесс без подтверждения сотрудника запускает массовое перебронирование. |
| T3 | Фактическая задержка не подтверждается. |
| T4 | Созданы лишние расходы и обращения. |

Первое расхождение: **T2**, прогноз используется как окончательное высоковлияющее решение.

Управляемые правила: прогноз меняет приоритет проверки; сохраняются вероятность и версия модели; массовое действие требует подтверждения ответственной роли.

### Раунд B — Контроль входных данных

| Такт | Подтверждённое событие |
|---|---|
| T0 | Модель обучена на завершённых рейсах и имеет принятую версию `M12`. |
| T1 | Для нового маршрута отсутствуют два обязательных признака, но запись допускается к прогнозу. |
| T2 | Модель возвращает `91%`, не показывая неполноту данных. |
| T3 | Рейс попадает в очередь массовых действий. |
| T4 | Сотрудник не может объяснить основание приоритета. |

Первое расхождение: **T1**, запись не прошла входной контроль, но отправлена в модель.

Управляемые правила: минимальная полнота обязательных признаков; фиксация версии модели и порога качества; неполные/внеобластные данные направляются сотруднику без автоматического действия.

### Источники и граница

BPMSoft позволяет настраивать и обучать ML-модели, использовать прогнозы в бизнес-процессах и хранить экземпляры модели. Конкретная модель задержки, набор признаков, пороги и процедура операционного подтверждения — **проектное решение**.

Основные источники: [модели машинного обучения](https://edu.bpmsoft.ru/baza-znaniy/nastroyka-ml-modeley/nastroennye-modeli-mashinnogo-obucheniya/?version=1.9), [обучение модели](https://edu.bpmsoft.ru/baza-znaniy/nastroyka-ml-modeley/nastroyka-i-obuchenie/?version=1.9), [процесс с ML-моделью](https://edu.bpmsoft.ru/baza-znaniy/nastroyka-ml-modeley/nastroyka-biznes-protsessa/?version=1.9).

## 11. Задание 45 — Кризисный центр мегахаба

### Цель

Доказать сквозную устойчивость одного набора правил при массовой задержке. Финал использует три контрольных прогона одного кризиса; уже исправленные правила сохраняются между раундами.

### Инициатор и кризис

Грозовой фронт снижает пропускную способность мегахаба. Погода объясняет массовое изменение расписания, но не объясняет откаты данных, дубли, лишние права и преждевременные уведомления. Лада Ветрова просит восстановить путь контрольного пассажира и его багажа, затем повторить тот же набор событий без скрытых нарушений.

### Раунд A — Единая версия маршрута

| Такт | Подтверждённое событие |
|---|---|
| T0 | Система расписания публикует версию `12` задержанного сегмента. |
| T1 | Пассажирский кейс и багаж получают маршрут версии `12`. |
| T2 | Поздняя версия `11` принимается и возвращает старое время/выход. |
| T3 | Багаж продолжает старый маршрут. |
| T4 | Пересадка помечается доступной по устаревшим данным. |

Первое расхождение: **T2**.

Конфигурация: единый стабильный ID сегмента; принимать только более новую версию; события багажа проверять в контексте актуального сегмента; отклонение сохранять в журнале.

### Раунд B — Партнёр и ответственное решение

| Такт | Подтверждённое событие |
|---|---|
| T0 | Для партнёра открывается временный доступ к пассажирам его сегмента. |
| T1 | Реестр визуально отфильтрован по организации партнёра. |
| T2 | По прямой ссылке партнёр открывает чужой кейс и финансовые колонки. |
| T3 | Автоматизация формирует дорогой партнёрский вариант без задачи сотруднику. |
| T4 | Пассажиру готовится неподтверждённое изменение. |

Первое расхождение: **T2**. T3–T4 остаются обязательными критериями контрольного прогона, но не заменяют первый checkpoint.

Конфигурация: права по организации/записи и колонке; отрицательный тест прямой ссылки; высоковлияющий вариант требует подтверждения сотрудника; финальное сообщение ждёт подтверждения партнёра.

### Раунд C — Наблюдаемый внешний результат

| Такт | Подтверждённое событие |
|---|---|
| T0 | ML-прогноз повышает приоритет кейса; сотрудник подтверждает план восстановления. |
| T1 | Процесс отправляет бронирование с `correlationId=C-990`, `operationKey=RB-990`. |
| T2 | Наступает таймаут; система помечает операцию успешной, не зная результата. |
| T3 | Пассажиру отправляется подтверждение. |
| T4 | Повтор с новым ключом создаёт вторую бронь. |
| T5 | Поддержка не может связать уведомление, процесс и обе операции. |

Первое расхождение: **T2**, неизвестный исход ошибочно превращён в успех.

Конфигурация: таймаут → наблюдаемое состояние `неизвестно`; повтор с тем же operation key и сохранённой корреляцией; уведомление только после подтверждённой внешней брони; журнал связывает рейс, кейс, процесс и команду.

### Финальная приёмка

Успешный прогон обязан одновременно доказать:

- одна актуальная запись операционного сегмента;
- ни одно старое или повторное событие не меняет актуальное состояние;
- пассажирская ситуация остаётся одним кейсом;
- партнёр не читает чужие записи и закрытые колонки;
- высоковлияющее решение подтверждает сотрудник;
- прогноз используется как рекомендация/приоритет, а не автоматический приговор;
- внешняя операция не считается успешной без подтверждения;
- пассажир получает ровно одно согласованное уведомление;
- журнал позволяет пройти путь от входного события до бизнес-результата по одному correlation ID.

Основные источники: официальные страницы BPMSoft, перечисленные для заданий 37–44. Авиационные идентификаторы, версии событий, ключи идемпотентности и финальные бизнес-пороги остаются проектными решениями.

## 13. Ворота перед техническим прототипом

Этап 2 считается завершённым, если:

- для каждого раунда есть ровно один доказуемый первый checkpoint;
- поздний симптом визуально заметнее, но логически не подменяет checkpoint;
- управляемых правил не больше трёх;
- позитивный прогон проверяет не только финальный статус;
- проектные решения явно отделены от возможностей BPMSoft;
- механика не сводится к выбору карточки: игрок обязан наблюдать, остановить/отмотать, настроить и повторить;
- соседние задания различаются основной границей знания;
- финал использует ранее изученные правила и не вводит новый правильный ответ.
