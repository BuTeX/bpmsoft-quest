# Пятая карта — прототип 37A и ранняя техническая проверка

Актуально на 2026-07-21. Статус: исторический протокол завершённого Этапа 4; production-результат описан в [`CHAPTER_5_FINAL_QA.md`](CHAPTER_5_FINAL_QA.md).

## 1. Что реализовано

Изолированный вертикальный срез не подключён к основному прогрессу и серверному состоянию:

- `chapter5-simulation.js` — чистый детерминированный движок;
- `chapter5-prototype-data.js` — конфигурация задания 37, раунда A;
- `chapter5-prototype.html` — отдельный UX-стенд;
- `chapter5-prototype.css` — временная облачно-небесная графика в палитре GoodProgram;
- `chapter5-prototype.js` — управление временем, журналом, checkpoint, правилами и повторным прогоном;
- `test-chapter5-simulation.mjs` — 11 тестов контракта и сценария.

Стенд намеренно не изменяет `index.html`, ключи сохранения и код карт 1–4. Для живого UX-теста его файлы добавлены только в публичный allowlist штатного сервера и доступны по `/chapter5-prototype.html`; основной прогресс и серверное состояние не затрагиваются.

## 2. Проверенный пользовательский путь

| Шаг | Наблюдаемый результат |
|---|---|
| Исходный прогон | T0–T4 воспроизводятся автоматически; журнал и сцена показывают один такт |
| Дефект | на T3 версия 3 откатывает запись; на T4 уведомление использует 09:55 |
| Перемотка | после baseline доступны посещённые такты без изменения входных событий |
| Диагностика | T3 можно отметить как первый подтверждённый сбой |
| Ошибочная проверка | правильный T3 с исходным правилом даёт `rule-error`, попыток остаётся 3 |
| Обратная связь | показано первое оставшееся нарушение, но правильное значение настройки не раскрыто |
| Исправление | `higher-only` + `keep-with-reason`, составной ключ остаётся подтверждённым |
| Контрольный прогон | версия 3 отклонена, журнал сохраняет причину, уведомление использует 10:40 |

## 3. Автоматические проверки

Команда:

```bash
node --test test-chapter5-simulation.mjs
```

Результат: `11/11` тестов проходят.

Проверены:

- валидность конфигурации 37A;
- воспроизводимость исходного дефекта;
- глубокое равенство двух одинаковых прогонов;
- равенство прямого прогона и прогона после перемотки;
- правильная конфигурация и обязательные outcomes;
- отдельные ошибки checkpoint и правил;
- сохранение промежуточного forbidden outcome;
- переходы runtime baseline → diagnosis → verification → passed;
- запрет verification до завершения baseline и выбора checkpoint;
- отклонение неизвестных значений;
- отсутствие мутации данных миссии.

Полный `npm test` карт 1–4, прототипа и серверного/админского контура также проходит: `17/17` серверных тестов, `11/11` тестов operational twin и все проверки миссий 01–36 зелёные.

В стенд добавлен локальный UX-протокол. После успешного прохождения он показывает время baseline и всей сессии, паузы, перемотки, выбранные checkpoints, изменения правил, число контрольных прогонов и ошибок. Игрок может добавить две оценки и комментарий, затем скопировать текст. Данные существуют только в текущей вкладке и не отправляются на сервер.

## 4. Браузерная проверка

### Desktop `1440 × 1000`

- исходный и контрольный прогоны завершены через интерфейс;
- ошибочный и успешный outcomes визуально различаются;
- событие версии находится внутри сцены и следует по операционной линии;
- горизонтального overflow нет;
- ошибок и предупреждений консоли нет.

### Mobile `390 × 844`

- заголовок, четыре этапа, сцена, журнал и настройки переходят в одну колонку;
- ширина рабочей области и панелей `366 px` при viewport `390 px`;
- горизонтального overflow нет;
- пять тактов остаются отдельными кнопками;
- критические действия не требуют drag.

### Reduced motion

Для воспроизводимой QA-проверки стенд поддерживает `?reducedMotion=1` в дополнение к системному `prefers-reduced-motion`.

- переходы сокращаются до фактически мгновенной смены состояния;
- T0–T4 и пять записей журнала сохраняются;
- результат совпадает с обычным режимом;
- горизонтального overflow нет.

### Доступность управления

- все действия реализованы нативными `button` и `select`;
- временная шкала имеет групповую подпись и отдельные доступные имена тактов;
- текущее состояние сообщается через live region;
- фокус имеет контрастный `focus-visible`;
- цвет дублируется текстом и формой;
- визуальная сцена полностью продублирована журналом.

Автоматический браузерный backend сфокусировал кнопки, но не синтезировал нативную активацию Enter/Space. Поэтому реальное прохождение только клавиатурой остаётся обязательным наблюдением Этапа 5, хотя стенд использует стандартные элементы без нестандартной обработки клавиш.

## 5. Дефекты, найденные и исправленные во время проверки

1. Скрытая панель результата перекрывалась общим `display: flex` — добавлено нормативное правило `[hidden]`.
2. Кнопки временной шкалы ошибочно получали роль `listitem` — сохранена нативная роль `button`, контейнер стал `group`.
3. Маркер версии участвовал в обычном потоке вместо сцены — зафиксировано абсолютное позиционирование.
4. Grid-элементы на mobile расширяли страницу на один пиксель — добавлен `min-width: 0` для колонок workspace.

## 6. Решение по воротам

Технические ворота Этапа 4 пройдены:

- новая механика ощущается как временная диагностика, а не как карточный тест;
- checkpoint и правило проверяются независимо;
- повторный прогон действительно использует те же входные события;
- промежуточное нарушение нельзя скрыть хорошим финальным статусом;
- движок тестируется без DOM;
- desktop, mobile и reduced-motion состояния технически работоспособны;
- карты 1–4 не получили регрессий.

Исторический следующий шаг этого протокола выполнен в рамках сборки релиз-кандидата. Текущий статус карты и итоговые проверки находятся в [`CHAPTER_5_FINAL_QA.md`](CHAPTER_5_FINAL_QA.md); дополнительное исследование на нескольких пользователях теперь считается последующим улучшением.

Точный сценарий наблюдения и пороги: [`CHAPTER_5_UX_TEST_PROTOCOL.md`](CHAPTER_5_UX_TEST_PROTOCOL.md).
